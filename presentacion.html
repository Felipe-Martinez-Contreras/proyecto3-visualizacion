01El problema

Una autoridad pública necesita decidir en el momento: restricciones vehiculares, alertas ambientales, ajuste de semáforos inteligentes. Pero los datos de tráfico, aire y clima viven dispersos y sin relacionar.

🚗Movilidad

¿Dónde y cuándo se satura el tráfico? ¿Qué zonas están al límite?

🌫️Aire

¿La calidad del aire cruza umbrales críticos? PM2.5, NO₂, O₃.

🔗Relación

¿Más tráfico ⇒ más contaminación? ¿Cómo influye el clima?

Ciudad de ejemplo: Santiago · 3 zonas — Centro, Norte, Sur. El analista necesita ver el patrón, no leer tablas crudas.

02La solución

  • Tablero en vivo — se refresca solo cada 5 s, sin recargar la página.
  • 3 visualizaciones — tráfico en el tiempo, aire vs tráfico, clima vs tráfico.
  • 4 filtros interactivos — zona, rango de tiempo, métrica y estado de congestión.
  • Datos reales + simulados — clima y aire de Open-Meteo; tráfico simulado.
Usuario final: analista / autoridad pública. Una sola pantalla, decisión inmediata.

03Las 3 vistas y los 4 filtros

① Línea temporal

Vehículos vs tiempo. Cómo evoluciona la congestión de cada zona a lo largo del día.

② Scatter

PM2.5 vs vehículos. ¿Sube la contaminación con el tráfico? Cruce aire + tráfico.

③ Clima vs congestión

Temperatura vs tráfico. El clima global frente al volumen vehicular.

Filtros que tocan los datos

Zona (Centro/Norte/Sur) y Rango (5 min / 1 h / día) → se aplican en la consulta SQL. Congestión (bajo/medio/alto) → derivada al vuelo sobre el tráfico.

Filtro que toca la vista

Métrica (tráfico/aire/clima) → muestra u oculta cada panel sin re-consultar. Todos los filtros comparten un mismo callback.

05Stack tecnológico

🐍 Python 3.12 · uv
Un solo lenguaje de punta a punta. Entorno y dependencias con uv en un .venv aislado.
📊 Dash + Plotly
Backend Python + gráficos interactivos. Refresco en vivo con dcc.Interval y callbacks.
🐼 Pandas
Transformación y cruces. Todo el pipeline habla DataFrame: fuentes, repositorio y vistas.
🗃️ SQLite
Un archivo único db/monitoreo.sqlite, versionado en el repo y compartido por el equipo.
⏱️ APScheduler
BlockingScheduler para la ingesta continua: intervalo, coalesce, max_instances=1.
🐳 Docker + Compose
Dos servicios sobre una misma imagen, orquestados y compartiendo la BD por volumen.
Además: requests (clientes de API) y python-dotenv (config por .env). Dependencias pineadas por versión mínima, sin lockfile — proyecto de vida corta.

07Ingesta — un tick del ciclo

  • Orden fijo por tick: tráfico simulado → aire por zona → clima global.
  • Timestamp compartido por todas las filas del tick → coherencia temporal para los cruces.
  • Aislamiento de fallos: cada fuente en su try/except; si una cae (red), el tick sigue con las demás.
  • Timestamp inyectable → ejecución determinista, sin depender de la hora real.
def ejecutar_tick(timestamp=None):
    # todas las filas → mismo instante
    ts = timestamp or now()

    try:  # tráfico (sin red)
        df = simulador.generar_trafico(ts)
        repo.insertar_trafico(df)
    except Exception as e:
        log(e)  # registra y sigue

    for zona in ZONAS:  # aire por zona
        ...
    # clima global
    return {"trafico":3, "aire":3, "clima":1}

08Fuentes de datos — gratis, con fallback

clima

Open-Meteo. Temperatura, humedad, viento. Gratis, sin API key. Una coordenada (clima global).

aire

Open-Meteo Air Quality por defecto (sin key), con OpenAQ como fallback configurable. PM2.5, NO₂, O₃.

tráfico

Simulado. No hay API pública fiable. Ruido acotado ±25% sobre perfiles realistas por zona.

🔁 Fallback automático

Si la fuente primaria de aire lanza excepción (red/config), se intenta la otra. Ambas se normalizan al mismo esquema timestamp,zona,pm25,no2,o3 → intercambiables.

🎲 Simulación controlada

El Centro más congestionado, el Sur más fluido. RNG y timestamp inyectables → misma semilla, mismo resultado. Totalmente reproducible.

Decisión de equipo: se priorizó Open-Meteo Air Quality sobre el OpenAQ del enunciado para evitar registro/keys. Ambas gratuitas, mismos contaminantes objetivo.

10Patrones y decisiones de diseño

🧩Dos niveles: lógica pura vs runtime

En ingesta y dashboard, la lógica de datos (tick, consultas, figuras) se separa del runtime (scheduler, servidor Dash). Se puede ejecutar sin levantar procesos.

💉Inyección de dependencias

Timestamp y RNG inyectables por todos lados. Al fijar la hora y la semilla, la ejecución es determinista y reproducible: misma semilla, mismo resultado.

🗂️Repositorio (acceso a datos)

Una capa datos/repositorio.py concentra insert/consulta. El resto del código no escribe SQL. Cambiar SQLite→Postgres toca un solo lugar.

🛡️Consultas parametrizadas

WHERE con placeholders ? y params= — nunca f-strings con valores del usuario. Sin inyección SQL.

📐Esquema = única fuente de verdad

schema.sql define las tablas; los productores entregan DataFrames con esas columnas exactas → to_sql mapea por nombre, sin boilerplate.

🧮Datos derivados, no almacenados

La congestión es una función pura sobre vehículos/velocidad. Se calcula al leer → no hay estado que sincronizar.
